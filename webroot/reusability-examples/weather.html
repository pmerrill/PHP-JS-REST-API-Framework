MetaWeather Example

Info

This example uses the MetaWeather Location API and retrieves the forecast of a specific location. Looking at the source code will show you that this API call reuses a very large portion of the code I wrote for the RESTCountries challenge.

The MetaWeather API accepts a "location ID", which is set as a data attribute of the #api element.

Click the button above to get London's forecast.

app.js

MetaWeather was added to the source object in app.js. This is where the endpoint and parameters are defined as well as how the output is compiled and rendered.

The #api element's endpoint data attribute defines which MetaWeather path we use. This example defines the path as default, which means that we'll use what's defined in the source/MetaWeather/path/default object in app.js.

The default path sets the endpoint to /api/bonus-work/weather.php, creates the parameter(s), and sets up the resultsFactory.

When we render the results of the API call we use the resultsFactory. This lets us set a template for how the data gets compiled and rendered.

/api/bonus-work/weather.php

This is the endpoint the default MetaWeather path sets in app.js. It performs a curl request through a series of object-oriented abstractions.

/api/bonus-work/weather.php uses the APIController class and its subclasses to make curl requests, process the results, and output them in a universal format.

The APIController class exists in controllers/APIController.php and has handy $apiCall methods such as ->start(), ->setOptions(), ->execute(), ->end(), ->errorCheck(), and more.
